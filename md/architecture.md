# 1. **서비스 아키텍처**

## 1.1. 목적

기존의 미리 구축된 데이터셋 (AIHub, SNU등)과 뉴스 데이터를 지속적으로 수집, 가공하여 분석 모델을 학습시키고, 이 모델이 생성한 근거 데이터를 사용자에게 안정적으로 제공하는 전체 파이프라인입니다.

'모델 학습 환경'과 '사용자 트래픽 처리 환경’의 차이점을 인정하여, 서로 분리하여 운영 및 비용 효율성을 확보합니다.

## 1.2. 저장소 및 데이터

파이프라인의 흐름에 따라 생성되는 데이터을 정의하고, 각 목적에 맞는 저장소를 구성하여 데이터를 관리합니다.

- **데이터 정의**
    - **Raw Data:** 크롤링 직후 전처리를 거치지 않은 원본 뉴스 데이터
    - **Processed Data:** 텍스트 전처리 및 정제가 완료된 중간 데이터
    - **Dataset:** 외부 데이터 및 LLM 증강을 거쳐 최종적으로 구성된 모델 학습용 데이터셋
    - **Model Artifact:** 모델 학습이 완료된 후 도출된 가중치 및 모델 파일
    - **Inference Result:** 모델 추론을 통해 생성된 최종 결과물(뉴스 신뢰도 및 근거 데이터)
- **저장소 역할 명세**
    - **객체 스토리지 (AWS S3):** 크롤링 원본 데이터, 학습용 데이터셋, 그리고 학습이 완료된 모델 아티팩트를 저장하는 파이프라인의 주 저장소 역할을 수행합니다.
    - **관계형 데이터베이스 (AWS RDS):** 서비스 제공을 위한 최종 단계로, 근거가 포함된 추론 결과 데이터를 저장하여 백엔드가 활용할 수 있게 합니다.

## 1.3. 학습환경

- **자동화된 실행 파이프라인:** AWS EventBridge를 실행 주체로 설정하여, 크롤링 및 파이프라인 구성 요소들이 특정 주기마다 자동으로 반복 실행되도록 스케줄링합니다.
- **서버리스 기반 데이터 처리:** 뉴스 RSS 크롤링, 텍스트 전처리, 데이터 증강 등 간헐적이고 반복적으로 발생하는 작업 부하를 처리하기 위해 AWS Lambda 기반의 서버리스 아키텍처를 채택하여 유휴 자원 비용을 최소화합니다.
- **Redis를 활용한 중복 데이터 방지:** 크롤링 단계에서 인메모리 캐시 시스템인 Redis를 사용하여 뉴스 기사의 URL이나 제목을 해싱하여 저장합니다. 이를 통해 기사의 중복 수집 여부를 빠르게 판단하고 불필요한 데이터 처리 비용을 절감합니다.
    - Elascticache 고려(비용, 운영 관점에서)
- **원본 데이터 영구 보존:** 크롤링 이후 전처리를 수행하기 전의 원본 데이터를 S3에 별도로 저장합니다. 이는 향후 새로운 목적의 파이프라인 확장을 위한 유연성을 확보하고, 파이프라인 디버깅 및 역추적을 용이하게 합니다. 또한 전처리 로직이 변경되었을 때 과거 데이터를 온전히 재처리하기 위해 사용됩니다.
- **LLM 기반 학습 데이터 증강:** 기존에 구축된 외부 데이터셋은 주로 '낚시성 뉴스'에 편중되어 있는 한계가 있습니다. 이를 해결하기 위해 실제 정상 뉴스 데이터를 수집한 후, 외부 LLM을 활용하여 '가짜 뉴스' 학습 데이터를 자체 생성합니다.
- **이원화된 모델 저장소 운영:** 학습이 완료된 모델은 내부 백업 및 파이프라인 연동을 위해 AWS S3에 적재하며, 외부 배포 및 체계적인 서비스 버전 관리를 위해 Hugging Face Hub를 병행하여 사용합니다.

## 1.4. 운영환경

- **배치 기반 비동기 추론:** 사용자 요청이 발생할 때마다 실시간으로 모델을 구동하는 대신, 하루 2~3회 정해진 주기에 따라 일괄적으로 데이터를 분석하고 추론을 수행합니다. 이를 통해 고비용의 실시간 GPU 연산 자원 대신 최적화된 인스턴스를 활용할 수 있어 컴퓨팅 비용을 대폭 절감합니다.
- **읽기 전용 서빙 구조:** 모델이 분석을 마친 최종 근거 데이터는 서비스용 관계형 데이터베이스(AWS RDS)에 사전 적재됩니다. 사용자 요청을 처리하는 백엔드 서버(AWS EC2)는 무거운 머신러닝 연산 없이 RDS에서 정형화된 데이터를 단순 조회하여 응답하므로, 지연 시간을 최소화하고 서비스 가용성을 높입니다.

## 1.5. 모니터링 및 로깅

- **중앙 집중형 로그 관리:** **Amazon CloudWatch Logs**를 통해 전체 시스템의 로그를 통합 관리합니다.
- **실시간 지표 및 알람:** 시스템 리소스 사용량과 파이프라인 성공 여부를 모니터링합니다. 작업 실패나 비정상적인 지표 감지 시 **CloudWatch Alarms**를 통해 관리자에게 디스코드 알림을 발송합니다.
- **비용 및 성능 최적화:** 로그 보존 정책을 설정하여 불필요한 로그 데이터 저장을 방지하고, **CloudWatch Logs Insights**를 사용하여 대량의 로그 데이터에서 특정 에러 패턴을 신속하게 검색합니다.
- **통합 시각화 대시보드 구축 (Grafana):**  CloudWatch의 인프라 메트릭과 비즈니스 지표를 하나의 화면에서 통합 모니터링합니다.

## 1.6. 장애 대응

### 1.6.1. 크롤링 실패

- **장애 시나리오:** 타겟 언론사 RSS 피드의 일시적 응답 지연, 네트워크 타임아웃, 또는 대상 웹사이트의 HTML 구조 변경으로 인한 파싱 에러
- **감지 메커니즘:** AWS CloudWatch를 통해 Lambda 함수의 실행 에러 및 타임아웃 지표를 모니터링하여 임계치 초과 시 관리자에게 알람을 발송합니다.
- **대응 및 복구 전략:**
    - **재시도 및 격리:** 일시적인 네트워크 오류를 대비하여 Lambda 자체의 비동기 재시도 기능을 2회로 설정합니다. 최종 실패한 이벤트 페이로드는 데드 레터 큐(DLQ, Amazon SQS)로 전송하여 파이프라인 전체의 중단을 방지합니다.
    - **중복 방지:** 크롤링이 실패하여 수동으로 재실행하더라도, 기존에 구축된 Redis 해시 캐시를 통해 이미 수집된 기사는 필터링되므로 데이터 중복 적재를 원천 차단합니다.
    - **구조 변경 대응:** 웹사이트 구조 변경으로 인한 파싱 에러의 경우, 다시 코드를 수정하여 배포한 후 원본 데이터(S3의 Raw Data)를 기반으로 재처리를 수행합니다.

### 1.6.2. Spot interruption → 람다 쓰면 신경 안 써도 되긴 함

- **장애 시나리오:** 비용 절감을 위해 AWS SageMaker Managed Spot 인스턴스 또는 EC2 Spot 인스턴스를 활용하는 과정에서, AWS의 자원 회수 정책에 의해 인스턴스가 강제로 종료되는 상황
- **감지 메커니즘:** AWS EventBridge를 통해 Spot Instance Interruption Notice(2분 전 경고 알림) 이벤트를 감지합니다.
- **대응 및 복구 전략(솔직히 이건 맞는지 판단이 안섬. 모델 학습안해봐서 모르겠음):**
    - **학습 환경:** 체크포인트(Checkpointing) 기능을 구현합니다. 모델 학습 중 일정 에포크(Epoch)나 스텝 단위로 모델의 가중치 상태를 S3에 정기적으로 저장합니다. Spot 인스턴스가 회수된 후 새로운 인스턴스가 할당되면, 처음부터 다시 학습하는 것이 아니라 S3에 저장된 최신 체크포인트를 불러와 이어서 학습(Resume Training)을 진행합니다.
    - **서비스 환경 (배치 추론):** 처리해야 할 데이터 목록을 상태 값(대기/진행중/완료)과 함께 관리합니다. 추론 중단 시 미완료된 데이터 배치를 식별하고, 가용 영역(AZ)이나 인스턴스 타입을 변경한 새로운 자원을 프로비저닝하여 남은 작업을 자동으로 재개합니다.

### 1.6.3. 외부 API 호출 실패 (이건 이 문서에 쓸건 아닌듯 싶음. 그냥 구현이라)

- **장애 시나리오:** 외부 LLM을 호출하여 가짜 뉴스를 생성하는 Lambda 단계에서, API 서비스 자체의 장애가 발생하거나 호출 빈도 제한에 걸려 응답을 받지 못하는 상황.
- **감지 메커니즘:** Lambda 내부의 예외 처리 로직에서 HTTP 상태 코드 및 에러를 포착하여 CloudWatch 로그로 남깁니다.
- **대응 및 복구 전략:**
    - **지수적 백오프 (Exponential Backoff):** API 호출 빈도 제한이나 일시적 부하를 해결하기 위해, 재시도 간격을 기하급수적으로 늘려가며 호출하는 지수적 백오프 알고리즘을 코드 내에 구현하여 재시도합니다.
    - **서킷 브레이커 패턴 및 지연 처리:** 외부 API의 완전한 서비스 중단이 확인될 경우, 계속해서 호출하여 컴퓨팅 자원을 낭비하지 않도록 서킷 브레이커(Circuit Breaker)를 작동시킵니다. 처리하지 못한 뉴스 텍스트 원본은 큐(SQS)에 안전하게 보관하고, 외부 API가 복구된 이후에 큐에 쌓인 메시지를 소비하여 누락 없이 데이터 증강 파이프라인을 마무리합니다.
